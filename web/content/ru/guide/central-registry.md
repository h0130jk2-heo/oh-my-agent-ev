# Руководство: Центральный реестр

## Обзор

Модель центрального реестра использует репозиторий oh-my-agent на GitHub (`first-fluke/oh-my-agent`) как версионированный источник артефактов. Проекты-потребители загружают конкретные версии навыков и рабочих процессов, обеспечивая согласованность между командами.

Подход корпоративного уровня для организаций, которым нужно:
- Фиксация версий в нескольких проектах
- Аудируемые обновления через PR
- Проверка контрольных сумм SHA256
- Автоматические еженедельные проверки обновлений
- Ручная проверка перед применением обновлений

---

## Архитектура

Центральный реестр (first-fluke/oh-my-agent) содержит: release-please workflow, CHANGELOG.md, GitHub Releases с тарболами и контрольными суммами. Проекты-потребители настраивают `.agent-registry.yml`, `check-registry-updates.yml` и `sync-agent-registry.yml`.

---

## Для мейнтейнеров: Выпуск новых версий

oh-my-agent использует [release-please](https://github.com/googleapis/release-please). Conventional commits на `main` -> release-please создаёт PR с бампом версии и CHANGELOG -> слияние PR -> Git-тег + GitHub Release -> CI строит тарбол + SHA256 + prompt-manifest.json.

| Префикс | Бамп версии |
|---------|------------|
| `feat:` | Minor (1.x.0) |
| `fix:` | Patch (1.0.x) |
| `feat!:` или `BREAKING CHANGE:` | Major (x.0.0) |
| `chore:`, `docs:`, `refactor:` | Без бампа |

## Для потребителей: Настройка проекта

### .agent-registry.yml

```yaml
registry:
  repo: first-fluke/oh-my-ag
version: "4.7.0"

auto_update:
  enabled: true
  schedule: "0 9 * * 1"  # Понедельник 9:00 UTC
  pr:
    auto_merge: false
    labels: ["dependencies", "agent-registry"]

sync:
  target_dir: "."
  backup_existing: true
  preserve:
    - ".agent/config/user-preferences.yaml"
    - ".agent/config/local-*"
```

### check-registry-updates.yml

Проверяет новые версии по расписанию, создаёт PR при обнаружении обновления. Не создаёт дубликаты.

### sync-agent-registry.yml

Скачивает артефакты, проверяет SHA256, бэкапит `.agents/`, распаковывает, восстанавливает сохранённые файлы, коммитит.

## Сравнение: Центральный реестр vs GitHub Action

| Аспект | Центральный реестр | GitHub Action |
|--------|-------------------|--------------|
| **Сложность настройки** | Выше (3 файла) | Ниже (1 workflow файл) |
| **Контроль версий** | Явная фиксация | Всегда обновляет до последней |
| **Проверка контрольной суммы** | Да — SHA256 | Нет |
| **Откат** | Смена версии в `.agent-registry.yml` | Реверт коммита обновления |
| **Аудит** | PR с фиксированными версиями и лейблами | История коммитов |
| **Сохранение файлов** | Настраиваемые glob-паттерны | Встроенное: `user-preferences.yaml`, `mcp.json`, `stack/` |
| **Источник обновлений** | GitHub Release (тарбол) | npm-реестр |
| **Воздушный зазор** | Возможно — скачать тарбол вручную | Требуется доступ к npm |

## Когда какой использовать

### Центральный реестр когда:

- Несколько проектов на одной версии
- Нужен аудит обновлений с контрольными суммами
- Политика безопасности требует явного одобрения зависимостей
- Нужна фиксация версий с разными графиками обновления
- Air-gapped окружения

### GitHub Action когда:

- Один проект или немного независимых
- Простейшая настройка (один workflow файл)
- Устраивает автоматическое обновление до последней версии
- Встроенное сохранение конфигурационных файлов
